# ETF 投资研报 Agent

输入 ETF 代码或名称，获得一份结构化技术分析研报。支持追问、实时行情、多源降级。

三种模式（方向键菜单进入）：
- **单只深度研报** — 完整 11 维技术分析 + 可追问
- **批量快筛** — 从自选池挑多只，只出「评分 + 空仓/持仓建议」，一眼扫完一池标的，省 token
- **管理自选池** — 分组维护批量快筛的标的池

**核心原则：本地能做的不花 LLM 的钱。**

指标计算全部在本地完成（均线、MACD、RSI、KDJ、布林带等），LLM 只负责综合判断和自然语言解读。

## 核心设计决策

### 1. LLM 是分析师，不是计算器

所有技术指标（MA/MACD/RSI/KDJ/布林带/ATR/回撤等）在本地用 pandas 计算完毕，只把结论性数据（数值+判断标签）传给 LLM。LLM 负责综合多指标的矛盾信号做出判断，并生成人话解读。

**为什么：** 指标计算是确定性的，用 LLM 算既贵又不可靠。LLM 的价值在于"RSI 超卖 + MACD 绿柱缩短 + 缩量"这种多信号综合权衡。

### 2. 输入解析三级降级

```
纯数字 "512980"  → 直接验证（0次 LLM）
名称 "传媒ETF"   → 本地模糊匹配（0次 LLM）
模糊 "传媒那个"  → LLM 提取关键词 → 再匹配（1次 LLM，deepseek-chat）
```

能本地解决的不调 LLM，必须调时用最便宜的 deepseek-chat。

### 3. 数据源三级降级 + 缓存

按 ETF 独立文件缓存。缓存已含今天时秒返回，不请求任何 API；缓存到昨天时逐个数据源尝试增量，直到拿到新数据为止（不因某个源"返回成功但数据旧"而跳过后续源）。三源全挂还有过期缓存兜底。

日K缺今天时，用新浪/腾讯实时接口补上。接口自带收盘状态标记（新浪 field[32]、腾讯时间戳），已收盘则直接写入缓存，不依赖本地时钟判断。

这个收盘状态（`settled`）还会透传给指标计算：盘中（未收盘）时今日成交量只是累计到此刻的快照，拿去和过去 5 天的全天均量比会被系统性低估，所以会给量比加"盘中累计仅供参考"的警示，避免 LLM 把半天的量误判成缩量。

### 4. 两套 Prompt 分治

研报和追问用独立的 System Prompt：
- 研报 Prompt：严格约束 JSON 格式 + 11 项结构化输出
- 追问 Prompt：自由文本，streaming 逐字输出

一套 Prompt 加条件分支会互相干扰——格式约束和自由回答是矛盾的。

### 5. 智能选择：缓存置顶 + 代码直选

多个匹配结果时，有本地缓存的 ETF（即用户之前看过的）优先展示在"猜你想看"区域。选择时既支持输入序号也支持直接输入 6 位代码，减少用户的选择成本。

### 6. 延迟加载：秒开启动

`akshare`、`anthropic`、`openai` 等重型依赖不在启动时 import，而是延迟到实际调用时加载（Python `sys.modules` 保证只加载一次）。ETF 列表优先读本地缓存，缓存有效时完全不触发 akshare import。启动时间从 5s+ 降到 ~1s。

### 7. 变号指标不能用乘法阈值

MACD 背离判断里 DIF（`ema12 - ema26`）会在零轴上下变号。最初用 `dif < dif.max() * 0.8` 这种乘法阈值，当 DIF 为负时阈值方向会反转（`-0.01 * 0.8 = -0.008` 比 `-0.01` 还大），导致下跌段里价格一接近高点就误报"顶背离"。改成基于 DIF 自身振幅的加法容差（`dif < dif.max() - dif_range * 0.3`），符号安全。

**为什么：** 恒为正的量（价格、std）用乘法阈值没问题，会变号的量必须用加法/振幅归一化。这个 bug 是写单元测试时构造负 DIF 场景才暴露的——测试是审视代码的第二视角。

### 8. 操作建议按持仓状态分两个分支

这个 agent 是「分析」不是「记账」——它没有也不该有用户的持仓数据（是否持有、成本价、持仓量）。所以"建仓/加仓/持有/减仓/清仓"这种依赖持仓的词，用一个标签输出是没有依据的：对空仓和持仓的人含义往往相反。

操作建议因此拆成 `if_empty`（空仓时：建仓/观望）和 `if_holding`（已持有时：加仓/持有/减仓/清仓）两个分支，LLM 仍只基于技术面，但两类用户各取所需。

显示层对这两个分支做枚举校验：合法值原样显示，LLM 越界返回（如"梭哈"）会附 `⚠ 非标准值` 标记——不静默改写，而是标出来让用户能分清是预设建议还是 LLM 自由发挥。

**为什么不引入持仓输入：** 持仓/盈亏跟踪是有状态、要持久化的另一种职责，混进这个无状态的分析器会污染定位。真要做，应拆成独立的 sub-agent。

### 9. 操作建议要有风控纪律，而不是赌徒

最初 Prompt 没有任何风控约束，结果遇到全线空头、回撤 -22% 的标的，LLM 仍给"持有"——理由是"超卖了该反弹"，可它自己的风险提示里又写着"超卖可以更超卖"，动作和风险自相矛盾，本质是在赌反弹。

加了一套**趋势止损纪律**：确认下跌趋势（跌破全部均线+空头排列+MA20 下探）且无企稳信号时，持仓建议倾向减仓保本；超卖本身不构成持有理由；操作建议必须与风险提示自洽。

但纪律本身也要克制，避免从一个极端走到另一个：
- **跌幅是沉没成本，不是决策依据**：已经跌掉的回撤属于过去，"跌太多该止损"和"跌太多舍不得割"是同一个锚定错误的两面。决策只看前瞻信号（趋势/动能/有无企稳）；回撤只用来衡量当前风险高低，不当买卖扳机。
- **回撤阈值不能一刀切**：高波动行业 ETF（看 `atr_pct`）正常回撤就比宽基大，用固定 -15% 套所有标的会误伤。
- **震荡不等于下跌**：缠绕震荡可能是筑底，不适用保本减仓条款，否则又变成另一种"赌它还跌"。
- **减仓也讲执行节奏**：贴着支撑别市价砸出，逢反弹到 MA5/MA10 分批减——把"该减"和"别割在地板上"调和。
- **诚实交代局限**：这工具只有技术面一只眼，看不到估值、基本面、持仓成本、期限、仓位占比与集中度（多只同类高 beta 是押同一个风险）。机械地在超卖区减仓，对长期定投者可能恰恰是"卖在地板上"。所以每份研报底部都有免责脚注。

**为什么：** 没有纪律的"分析"就是掷骰子；但纪律定得太死又会从赌徒变成另一种刻舟求剑。专业的做法是有纪律、按品种波动调整、并诚实交代自己看不到什么。

### 10. LLM 的 JSON 输出要能自愈

LLM 偶尔会在字符串值里写未转义的英文双引号（如 `"信号仅为"弱转强"而非买入"`），直接破坏 JSON，导致整份研报解析失败、前面拉数据算指标的功夫全白费。

三层防护：① **DeepSeek JSON Mode**（`response_format={"type":"json_object"}`）从 API 层硬保证输出是语法合法的 JSON——这是 OpenAI 格式特性，所以研报这步用 OpenAI SDK 走 OpenAI 端点（`api.deepseek.com`），不能用 Anthropic 端点（实测该端点会静默忽略此参数）；② Prompt 里要求字符串内部引用一律用中文引号「」，双保险；③ 万一仍解析失败，不直接放弃，把坏 JSON 丢给**快模型（deepseek-chat）**修一次再解析——补转义引号是纯机械活，用便宜的快模型即可。

**为什么：** JSON Mode 把"未转义引号破坏 JSON"这类头号故障从源头掐掉，修复器从"经常要用"降级成"几乎用不上的保险"。但 JSON Mode 只保证语法合法、不保证字段齐全，所以消费端（`format_report`）对每个字段都用 `.get(..., "?")` 容缺渲染、关键字段缺失标 `?` 而非崩溃——硬约束 + 容缺渲染，比靠重试补字段更可靠（金融场景标 `?` 比瞎编一个值安全）。

### 11. 批量快筛只裁"输出"，不裁"脑子"

最初为省钱把批量换成了 `deepseek-chat`（不走思考链）+ 砍掉逐维度输出。结果遇到「多头排列但严重超买」的标的（如 512480：RSI 74、突破布林上轨），快筛给 8.5 分、建议"建仓/持有"，点进深度研报（reasoner）却是 6.5 分、"观望/减仓"——**同一只标的两套结论，偏差还系统性偏乐观**。

挖下去是两个原因叠加：① **模型不同**——reasoner 能做"趋势虽强但已过热，要对回调打折"的二阶推理，chat 倾向看面值；② **Prompt 结构不同**——研报强制逐项输出 `momentum/bollinger/risk/score.breakdown`，这个结构本身就是 chain-of-thought 脚手架，逼模型把超买风险写出来、折进评分（它的 breakdown `趋势9+动量7+量价5+风险5` 平均正好 6.5，是风险维度把分压下来的）；lite 砍了这套结构，等于抽掉了"先权衡风险再打分"的脚手架。还有个盲区：风控纪律（决策 #9）主要防"下跌赌反弹"，**管不到"上涨追高"**，所以超买情形两个模型各自发挥。

**修法不是给 lite 打补丁，而是让两边用同一套脑子：批量改用和深度研报同一个模型（reasoner）、同一套分析纪律，只把输出从 11 项裁成 4 个汇总字段。** 模型的思考过程是内部的、不算"返回内容"，所以"批量只返回那几个字段"依然成立。改完同一只 512480：快筛 6.0 分、"观望/持有"，理由也点出"超买+突破上轨追高风险大"——和深度研报对齐了。

**代价要诚实交代：** 批量不再是 chat 那么便宜了，reasoner 的思考 token 也计费（每只 completion 从 ~56 涨到 ~300）。但仍远小于完整研报（输出 11 项 ~1.5k token → 4 项），加上 lite 结论缓存（决策 #12）让盘后同日重复跑零 token。省 token 的杠杆从"换便宜模型"换成了"**裁输出 + 缓存**"——用一致性换掉了模型降级。

**lite Prompt 复用研报 Prompt 的「字段说明 + 风控纪律」内核**（从 `SYSTEM_PROMPT` 里切出来，单一来源不漂移），只换输出规范，并明确要求"内部照样逐维度权衡、只是不展开文字解读"。

**一个被否的替代：把多只塞进一个请求。** 实测 per-ETF + prompt 缓存后，塞一个请求只省 ~13% input，却放大"一个字段坏→整个 JSON 数组全军覆没"（决策 #10 的坑）、输出截断、标的串味、没法做进度条/单只重试。所以仍是 per-ETF 串行。

### 12. 双层缓存：数据缓存之外，再缓存 lite 结论

批量快筛会被一天跑好几次，所以在「数据缓存」之上又加一层「lite 结论缓存」，键 `code:date:settled:逻辑指纹`。盘后（`settled=True`）的结论当天不会变，第二次起同一池标的**零 token**——比省 input 缓存那点钱狠得多，且完全贴合"该走缓存走缓存"的哲学。

**键里为什么要带「逻辑指纹」：** 指纹是 `模型 + lite Prompt` 的短 hash（`lite_logic_version()`）。缓存键只有 `code:date:settled` 时踩过一个坑——把分析模型从 chat 换成 reasoner 后，同一只标的当天的旧 chat 结论还躺在缓存里被命中，"逻辑变了缓存却没失效"。加上指纹后，模型或 Prompt 一改，指纹就变、旧条目自动失效，不用手动清缓存。

**键里为什么要带 `settled`：** `settled`（今天这根 K 收没收盘）和 `stale`（数据到没到今天）是两回事。盘中的评分基于半天的量能快照，缓存了下次还是错的，所以**盘中结果根本不写缓存**（和数据层 `_save_cache` 只在收盘落盘完全一致），表格里还给盘中的标的打 `*` 角标提示仅供参考。为此 `calculate_indicators` 把 `settled` 显式导出到 `data_quality`，不再靠量比警示文案做 substring 匹配。

### 13. 自选池：分组 + 不可重建的状态要防丢

批量快筛的输入是一个分组的「自选池」（持仓/观察…），存 `cache/watchlist.json`——天然不进 git、不被 `sync_to_agent.sh` 覆盖、每个部署各管各的池子。但池子是**人工策展、不可重建**的状态（历史行情缓存能重建，池子不能），所以写入用「临时文件原子替换 + `.bak` 备份」双保险：写到一半崩了不会把池子写坏，误删主文件还能从 `.bak` 回退。

入池有四个入口，越顺手越好：研报看完顺手收藏（主路径）、管理菜单手动加、批量临时输入后回存、以及**冷启动从历史缓存导入**（`cache/` 里那些 `{code}_hist.json` 就是你之前看过的标的，一键灌进池子，零摩擦起步）。

### 14. 场外联接基金：技术面看 ETF，下单买联接，代码绝不猜

很多人不在场内买 ETF，而是在场外按净值申赎对应的**联接基金**（A/C 两类）。所以批量快筛表补了一列「联接基金(C)」：技术分析仍基于场内 ETF 的量价 K 线（联接只有每日净值、没有真实量价，对它做技术分析是误导），但表里直接给出你**实际要下单的那只场外联接代码**，默认显示 C 类（短线无申购费更划算）。

**关键约束：联接代码必须核对，绝不能让启发式或 LLM 猜**——买错代码=真金白银买错标的。所以映射是这样建的：akshare 拉全市场基金名录，按「ETF 的主题 + 管理人 + 含『联接』」筛候选，归类成 A/C 对——**只有唯一一对时才自动填，出现多对（如『创业板』会撞上创业板200/创业板50/创业板成长）一律让用户方向键人工选**（正解按 base 名最短排在第一位）。结果落 `cache/etf_feeder_map.json`，批量渲染只读这张核对过的表，离线零成本。没映射到的标的显示 `—`，不瞎填。

**为什么不实时匹配：** 名称启发式对干净标的够用、对歧义标的会选错，金融场景"选错"不可接受；而自选池是有限的几十只，一次性人工核对成本极低、收益是永久准确。这是"能本地做的不花钱"在数据正确性上的延伸——宁可让人确认一次，也不静默猜一个可能买错的代码。

## 技术栈

- **Python 3.12** + venv
- **AKShare** — 免费 A 股数据（ETF 列表、历史行情）
- **pandas** — 本地指标计算（rolling/ewm/diff/cummax 等时间序列操作）
- **DeepSeek** — LLM 调用。研报和**批量快筛同用 `deepseek-reasoner`**（思考型）+ JSON Mode 强制合法 JSON（OpenAI SDK），保证两边判断逻辑一致，快筛只是把输出裁成 4 个字段；关键词提取 / 追问 / JSON 修复用 `deepseek-chat`（便宜快）。批量靠 DeepSeek 自动 prompt 缓存把重复的 system prompt 摊薄
- **新浪/腾讯行情 API** — 实时数据 + 数据源降级
- **rich** — 批量快筛的表格渲染（按评分排序、配色）
- **simple-term-menu** — 方向键菜单 / 多选框（纯 termios、延迟加载、不破坏秒开）

## 快速开始

```bash

cd ~/etf-agent

# 2. 配置 API Key（写入 shell 环境，或稍后直接改 .env）

```

日常只需要记住一个命令：`./start.sh`。

脚本分工是：

- `start.sh`：主入口，给人用
- `run.sh`：实际启动器，固定使用项目内 `.venv`
- `init.sh`：环境初始化/重装依赖时手动用
- 不建议叫 `build.sh`：这个项目不是编译产物型项目，核心问题是“环境初始化”而不是“构建”

`start.sh` / `run.sh` 都会优先使用项目内 `.venv/bin/python`；若检测到还没初始化，会自动先跑一次 `./init.sh`。

### 同步到独立目录部署（sync_to_agent.sh）

`./init.sh` 会在缺少 `.env` 时自动从 `.env.example` 复制一份；从旧版本升级时，仍建议手动检查 `.env` 里的 key / base_url / 模型配置是否还是你要的值。若只是想单独初始化或重装依赖，再手动执行 `./init.sh`。

## 周4 学到了什么

1. **LLM 的角色是分析师而非计算器** — 确定性计算本地做，综合判断和人话解读交给 LLM，这是成本和可靠性的最优分工

2. **Agent 就是 Python 脚本 + while 循环** — 没有魔法框架，和 week1 的 chat.py 一样的交互模式，只是多了数据获取和指标计算

3. **数据源不稳定是现实问题** — 东方财富频繁限流，必须有降级方案；不同数据源成交量单位不同（股 vs 手），跨源拼接必须统一

4. **能本地做的绝不花 LLM 的钱** — 输入解析、代码匹配、指标计算全部本地，LLM 调用次数压到最少（代码输入只需 1 次）

5. **两套 Prompt 比一套加分支更干净** — 研报和追问的输出格式完全不同，硬塞一起会互相干扰

6. **单元测试能发现设计问题** — 写测试时暴露了死代码、注释过时、计算错误，测试是审视代码的第二视角

7. **自己用自己的产品是最好的需求来源** — 选择列表不支持直接输入代码、启动慢、匹配结果太多无从选择，这些都是实际使用中发现的痛点

8. **延迟加载是 CLI 工具的基本功** — 重型依赖放到实际调用时 import，用户不该为还没用到的功能付启动时间的代价

9. **省 token 要找对杠杆** — 批量快筛的省钱不在"把多只塞一个请求"（prompt 缓存早把重复 system prompt 摊薄了），而在"砍输出 + 换便宜模型 + 缓存结论"。一个一个串行请求看似笨，配合 DeepSeek prompt 缓存反而更省也更稳

10. **漏斗 + 聚焦** — 同一个分析能力拆成"批量快筛"（便宜、只出结论、扫一池）和"单只深度"（贵、全维度、可追问）两挡，让用户先低成本筛、再对值得的标的花钱细看，比"每只都出全量研报"的体验和成本都好
