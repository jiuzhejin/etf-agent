# Week4 Day1: 投资研报 Agent 架构设计

## 一、项目定位

独立的 CLI 程序，专注 ETF 投资分析。用户启动即进入分析模式，输入 ETF 代码或名称，获得结构化研报。

**核心原则：本地能做的不花 LLM 的钱。**

## 三、会话状态管理

本地维护当前会话状态：
- `current_code`: 当前分析的 ETF 代码
- `current_data`: 上一轮算好的指标数据
- `messages`: 对话历史（支持追问）

追问逻辑（本地判断）：
- 输入含新代码/名称 → 更新 current_code，重新拉数据计算
- 输入不含 → 复用 current_data，LLM 基于上下文重新解读

---

## 四、LLM 交互次数分析

| 场景 | LLM 调用次数 | 说明 |
|------|-------------|------|
| 输入代码 "512980" | 1次 | 本地直接算，只需 LLM 分析 |
| 输入名称 "传媒ETF" | 1次 | 本地模糊匹配到代码，再算+分析 |
| 输入模糊 "传媒相关的大基金" | 2次 | 第1次 LLM 解析意图得到代码，第2次分析 |
| 追问 "只看短线" | 1次 | 复用数据，LLM 重新解读 |
| 对比 "传媒和科技哪个好" | 2-3次 | 两个标的各自算好，LLM 对比分析 |

## 五、知识与数据分离

| 内容 | 放在哪 | 是否每次传 | 缓存 |
|------|--------|-----------|------|
| 角色定义 | system prompt | 是 | Prompt Caching |
| 指标含义（RSI>70=超买等） | system prompt | 是 | Prompt Caching |
| 输出格式约束 | system prompt | 是 | Prompt Caching |
| 指标数据 JSON | user message | 是 | 不可缓存（每次变） |
| 用户问题 | user message | 是 | 不可缓存 |

System prompt 固定不变 → Prompt Caching 命中后费用降 90%。

---

## 六、错误处理策略

| 错误类型 | 处理方式 |
|---------|---------|
| ETF 代码不存在 | 提示用户，建议检查代码 |
| 网络超时 | 等几秒重试 |
| 少量数据缺失（几天） | 跳过缺失点，正常计算，研报中标注 |
| 大量数据缺失 | 提示用户，不进行计算 |
| LLM 调用失败 | 重试，仍失败则只返回本地计算的原始指标 |

## 七、技术栈

- Python + AKShare（数据获取）
- 本地计算指标（pandas / numpy）
- Anthropic SDK（LLM 调用，streaming 输出）
- Structured Outputs（约束返回格式）
